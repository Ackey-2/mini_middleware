# Phase 6 RPC Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add first-class Service/Client discovery and synchronous request-response RPC APIs on top of the existing Pub/Sub data plane.

**Architecture:** Discovery gets `SERVICE` and `CLIENT` endpoint kinds and matches them by service name, request type, response type, and QoS reliability. Runtime RPC uses internal Pub/Sub topics with protobuf envelopes: clients publish `RpcRequest` to a service request topic, services publish `RpcReply` to the caller's private reply topic. `Node` owns created services and clients just like publishers and subscribers.

**Tech Stack:** C++17, CMake, Protobuf, GoogleTest, existing `Node`/`Publisher`/`Subscriber`/`DiscoveryAgent` APIs.

**Spec:** `docs/superpowers/specs/2026-06-24-phase6-rpc-design.md`

- [ ] **Step 3: Implement `Service<Req, Resp>`**

`Service` constructor receives service name, shared `LocalBus`, handler, and QoS. It creates a request `Subscriber<RpcRequest>`. On each request, parse `Req`, call handler, serialize `Resp`, publish `RpcReply` to the request's `reply_topic`.

- [ ] **Step 4: Implement `Client<Req, Resp>`**

`Client` creates a request `Publisher<RpcRequest>` and a reply `Subscriber<RpcReply>`. `call()` allocates a request id, inserts a pending waiter with `condition_variable`, publishes the request, waits until reply or timeout, erases the pending entry, and returns parsed `Resp` if `reply.ok()` and parse succeeds.

- [ ] **Step 5: Add `Node` factories**
